⚡ L2ARC Cache Config

ℹ️
L2ARC (Level 2 Adaptive Replacement Cache) is a read cache stored on a fast device (SSD or NVMe). ZFS automatically promotes hot data to L2ARC to reduce latency for repeated reads.

Best disk type: NVMe or high-endurance SSD. Capacity ideally ≥ your working data set (not RAM). Low-endurance SSDs will wear out quickly under heavy random read workloads.

Warning: Removing a cache device immediately evicts all its cached data — this does not cause data loss but may temporarily increase read latency.

Add Capacity — Guide
Understand what happens before you add disks
RAIDZ Expansion ZFS 2.3+ only · 1 disk
The selected disk is attached directly into your existing RAIDZ-1 or RAIDZ-2 vdev, increasing its capacity in-place. No new vdev is created — the pool's fault tolerance is fully maintained throughout the process. ZFS rebalances data in the background after expansion.

Why reported capacity may temporarily decrease: During expansion ZFS must rewrite existing data blocks to account for the wider stripe. Until rebalancing completes, ZFS reports the pool's usable capacity conservatively — it accounts for the worst-case space overhead of the in-progress rewrite. This is expected behaviour: reported capacity will rise progressively as each segment finishes rebalancing, returning to (and exceeding) the pre-expansion size once the process is complete. Pool data remains fully accessible and protected throughout.

⚠ Only available when your pool's top-level vdev is RAIDZ-1 or RAIDZ-2 and ZFS ≥ 2.3 is installed. Not available for mirror or stripe pools.
Adding a New Vdev Mirror · RAIDZ-1 · RAIDZ-2 · Stripe
A completely new, independent vdev is appended to the pool. ZFS stripes data across all vdevs — so capacity grows immediately. However, each vdev is independent: if the new vdev fails entirely, only the data on that vdev is lost (the rest of the pool survives).
💾
Mirror (min 2 disks)
All disks in the vdev are identical copies. Best read performance. Usable capacity = size of one disk × number of pairs.
💾
RAIDZ-1 (min 3 disks · RAID5)
Single distributed parity. Survives 1 disk failure. Usable = (n−1) × disk size.
💾
RAIDZ-2 (min 4 disks · RAID6)
Double distributed parity. Survives 2 simultaneous failures. Best protection for large-disk arrays.
💾
Stripe (any number of disks)
No redundancy within the vdev. Maximum usable space. Any disk failure in this vdev loses all data in it.
⚠ Important: Adding a new vdev is permanent — vdevs cannot be removed from a pool once added. All selected disks will have their existing data erased. Make sure you have selected the right disks before confirming.
